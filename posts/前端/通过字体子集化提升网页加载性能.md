## 引言

在前端性能优化中，字体经常是一个被低估的性能瓶颈。

对于中文网站来说，这个问题尤其明显：一个完整的中文字体文件动辄几 MB，而页面中真正用到的字，可能只有几百个。结果就是——字体下载时间长、首屏渲染被阻塞，LCP 指标被拖慢。

如果你不想接入复杂的字体服务，也不希望依赖第三方 CDN，那么**字体子集化**会是一个非常实用、性价比很高的解决方案。

---

## 使用前你需要知道的事

在开始之前，先简单确认一下你是否真的「适合」做这件事。

如果你的项目满足下面任意一点，那么继续往下看基本不会浪费时间：

- 项目中使用了自定义 Web 字体（TTF / OTF / WOFF 等）
- 字体文件体积较大，影响首屏加载速度
- 字体内容相对固定，不需要覆盖极大量的动态文本
- 不想引入在线字体服务或额外的运行时依赖

---

## 为什么字体子集化能提升加载性能

简单说一句话：**我们只加载真正会用到的字**。

完整中文字体往往包含上万个字形，而一个普通页面实际展示的内容却非常有限。字体子集化的过程，就是从原始字体中提取出「当前项目会用到的字符集合」，重新生成一个体积更小的字体文件。

在实际项目中，这通常能带来几个非常直观的变化：

- 字体文件从几 MB 降到几百 KB，甚至更小
- 字体更快下载完成，减少首屏阻塞
- LCP、FCP 等指标明显改善
- 用户在首屏看到「默认字体」的时间缩短

## 实践：使用 font-slice 生成字体子集

下面进入具体实操部分。

这里我选择使用 **font-slice** 这个工具，它的特点是：配置简单、侵入性低，非常适合在本地或构建阶段使用。

### 1. 安装依赖

在项目中安装 `font-slice` 作为开发依赖：

```bash
npm install --save-dev font-slice
# 或
yarn add -D font-slice
```

### 2. 编写生成脚本

在项目中创建一个简单的脚本文件，例如 `slice-font.js`：

```js
const createFontSlice = require('font-slice');
const path = require('path');

createFontSlice({
  fontPath: path.resolve(__dirname, 'YourFont.ttf'),
  outputDir: path.resolve(__dirname, './font-output'),
});
```

将需要处理的字体文件（例如 `YourFont.ttf`）放在这个脚本文件的同级目录下。

然后在终端中执行：

```bash
node slice-font.js
```

### 3. 查看生成结果

执行完成后，你会在 `font-output` 目录中看到以下内容：

- 一个 `font.css` 文件
- 多个被切分后的字体文件（通常是几十到上百个）

这些文件已经是处理好的「子集字体资源」。

---

## 在项目中接入子集字体

接下来就是把生成的字体真正用到项目中。

### 1. 复制字体资源

将 `font.css` 以及生成的所有字体切片文件，复制到你项目的静态资源目录中（例如 `public/fonts`）。

### 2. 引入字体样式

在项目入口样式文件或全局样式中，引入 `font.css`：

```css
@import './fonts/font.css';
```

或者根据你的项目结构，通过 `<link>` 的方式引入。

### 3. 应用到页面

按照 `font.css` 中定义的 `font-family`，将页面或主题字体切换为该字体即可。

到这里，字体子集化的接入就完成了。

---


## 总结

字体子集化并不是一个复杂的技术，但它往往能带来非常实在的性能收益。

如果你的项目正在被「字体文件太大、首屏加载慢」困扰，那么不妨花十几分钟试试这个方案。大多数情况下，你会发现这是一次非常划算的优化。
